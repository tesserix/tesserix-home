# tesserix-home → mark8ly migration matrix

This document inventories every `app/api/*` route in `tesserix-home` (the platform super-admin Next.js app at `tesserix.app`) and maps each one onto an endpoint in the new mark8ly monorepo (`services/auth-bff`, `services/platform-api`, `services/marketplace-api`, `services/otto`).

Routing today (legacy) is keyed on `lib/api/admin-fetch.ts` (`tesserix-home/lib/api/admin-fetch.ts:14-35`), which fans out to eight upstream services: `tenant`, `tickets`, `auth`, `settings`, `subscription`, `audit`, `feature-flags`, `notification`. None of those are services in the new mark8ly repo. The new mark8ly collapses everything into `platform-api` (tenant/store/onboarding/invitation/notification/auth-reset/location), `marketplace-api` (the very fat store-scoped admin: products, orders, returns, billing, notifications, audit-logs, branding, tickets, customers, etc.), `auth-bff` (sessions/MFA/handoff), and `otto` (support conversations).

The hard distinction this matrix surfaces: tesserix-home is a **platform super-admin** UI, while mark8ly's admin endpoints are nearly all **`/admin/stores/:storeId/*`** (per-merchant). Many tesserix-home features either have no equivalent (subscription stats, expiring trials, audit-logs cross-tenant aggregation, email-template CRUD, feature-flags, system-health) or are platform/IaC-only (Cloud Run, Cloud SQL, GCP Secret Manager, IAM, Cloudflare KV, GitHub Actions). The viable mappings are: tenant CRUD, audit-events ingest (write-only), and the auth-bff session/MFA surface.

## Summary

- **Total tesserix-home admin features inventoried: 60** (counting unique route+method combos as features; collapsing close siblings — e.g. all 5 `system-health` endpoints — into one row where they share a backend).
- **Mappable to new mark8ly: 11** (tenants, sessions/MFA, password-reset, OpenFGA check, audit-events ingest write-path, location lookups; each with caveats).
- **Platform-only / direct GCP-or-third-party (no mark8ly equivalent expected): 23** (IAM, Cloud Run, Cloud SQL, billing, logs, storage, pubsub, Cloudflare KV, GCP Secret Manager, GitHub Secrets, releases/*, system-health, contact).
- **Gaps (legacy endpoint exists, no new equivalent yet): 26** (subscriptions admin/stats/plans/trials/portal, email-templates CRUD + notifications, feature-flags + experiments, tickets cross-tenant aggregation, audit-logs cross-tenant aggregation, content pages, onboarding-content CMS, system-health, openfga store listing, contact form).

## Decisions surfaced

These are the ambiguities and forks that need a human call before any per-feature PR can land. Each is concrete and actionable.

1. **Tenant lifecycle: where do super-admin destructive operations live?** New mark8ly's `services/platform-api/internal/tenant/handler.go` has only GET/PATCH and a getMe — no create, no delete, no batch operations. The legacy tesserix-home wires up `POST /tenants`, `DELETE /tenants/:id/admin-delete`, `batch-delete`, `delete-all`, `deletion-preview`. **Decision:** either (a) add a platform-admin-only `/internal/tenants/admin/*` group on platform-api, or (b) drop these UIs from tesserix-home and force tenant lifecycle through ops scripts. Per the project's "minimize platform-api surface" stance in the new tenant package comment ("Tenant creation is NOT exposed here — tenants are created exclusively through the onboarding completion flow"), option (b) seems intended — but the super-admin needs *some* destructive path.

2. **Tenant-scope vs store-scope for super-admin views.** Many tesserix-home features are tenant-keyed: subscription, audit-logs, tickets, content. Mark8ly almost universally moves to store-scoped (`/admin/stores/:storeId/*`). A tenant can have N stores, so any super-admin view across "this tenant" needs a fanout-across-stores layer or a new platform-api projection. **Decision:** pick one of (a) add `services/platform-api/internal/admin/*` aggregate endpoints that join across stores, (b) accept the per-store UX and add a store-picker to every super-admin page, (c) accept "first store only" as a heuristic in tesserix-home.

3. **Plans + Stripe settings + cross-tenant subscription stats.** None of these exist in mark8ly: `/api/v1/plans*`, `/api/v1/admin/stats/*`, `/api/v1/admin/settings/stripe*`, `/api/v1/admin/expiring-trials`, `/api/v1/admin/invoices`. Mark8ly hardcodes plans (`internal/billing/pricing/`) and uses GCP Secret Manager / env vars for Stripe creds. **Decision:** are these UIs being dropped (PRs to remove from tesserix-home) or rebuilt (significant new endpoints + DB tables in marketplace-api)?

4. **Email templates and notifications history.** The new mark8ly platform-api `notification` package is internal-only (constants, no HTTP CRUD). Tesserix-home expects a full template-CRUD + sent-email-log surface. **Decision:** rebuild this in platform-api (new tables + handlers + FGA roles), or drop the UI? If rebuilding, scope: should it live in `platform-api/internal/notification/` (single-tenant model) or in `marketplace-api` (per-store templates)?

5. **Feature-flags, system-health, onboarding-content (CMS), contact form, and the legacy go-shared release flow.** Five smaller features with no mark8ly equivalent. Each is independent. **Decision per feature:** keep (rebuild) / drop (delete from tesserix-home). For onboarding-content specifically, the source-of-truth question — is the marketing CMS even in scope for the new mark8ly admin? — needs answering before any wiring decision.
